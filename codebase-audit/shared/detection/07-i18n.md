# Detection Spec 07 — Internationalization

**ISO 25010:** Portability (Adaptability)
**Audit key:** `i18n`
**Static analysis only.** Two tiers: **universal correctness** (real bugs even single-locale)
and **localization readiness** (advisory unless the app is user-facing multi-locale).

## How to use this spec (agent instructions)

1. Read `discovery_context`. Determine **i18n intent**: is there an i18n framework
   (`gettext|i18next|react-intl|vue-i18n|babel|ICU|django.utils.translation`), locale files
   (`locale/|*.po|messages/|translations/`), or user-facing rendered text?
2. **Universal-correctness items (encoding, timezone, charset)** always apply to matching
   archetypes.
3. **Localization-readiness items** apply only if i18n intent exists OR the archetype is
   user-facing. If no i18n intent and single-locale-by-design → emit those as `skipped`
   (or Info), do NOT spam findings.
4. Run `signals`, read candidate snippets, apply `confirm`, assign `severity`.
5. Consult `examples/07-i18n.md` only when ambiguous.
6. Emit `AuditResult` JSON per `shared/schema.json`. Score per `shared/scoring.md`.

**Never flag non-user-facing text:** logs, debug messages, exception text, internal enum
keys, machine protocol strings need no translation.

## Universal Correctness (apply broadly)

### i18n_encoding — Consistent UTF-8, explicit encoding
- **intent:** Implicit/locale-default encodings corrupt accented and non-Latin text and cause
  `UnicodeDecodeError` in production.
- **applies_to:** all backend, fullstack
- **signals:** `open\([^)]*\)` without `encoding=`; `.encode\(\)/.decode\(\)` without an
  explicit codec; `.encode\(['\"](ascii|latin-1)`; `# -*- coding` legacy; byte/str mixing.
- **confirm:** warning if file/byte I/O relies on the platform-default encoding instead of an
  explicit UTF-8. pass if UTF-8 is explicit and consistent.
- **severity:** Medium (holds at all maturities — correctness)
- **remediation:** Pass `encoding="utf-8"` explicitly; standardize UTF-8 across the stack.
- **compliance_refs:** ISO25010 Adaptability; W3C i18n

### i18n_timezone — Timezone-aware datetimes, UTC storage
- **intent:** Naive datetimes and local-time storage cause off-by-hours bugs, DST errors, and
  wrong comparisons across regions.
- **applies_to:** all backend, fullstack, data-ml
- **signals:** `datetime\.now\(\)(?!.*tz)|datetime\.utcnow\(\)|date\.today\(\)` used for
  stored/compared timestamps; `new Date\(\)` stored as local; missing `USE_TZ`/tz config.
- **confirm:** fail if timestamps are stored/compared as naive/local time. pass for
  tz-aware UTC storage with localization at display time. (Durations/monotonic timing exempt.)
- **severity:** Medium (holds — correctness)
- **remediation:** Use tz-aware UTC (`datetime.now(timezone.utc)`); localize only at display.
- **compliance_refs:** ISO25010 Adaptability

### i18n_charset_storage — Full-Unicode database charset
- **intent:** `utf8` (3-byte) in MySQL rejects emoji and some scripts; columns must use
  `utf8mb4` (or an equivalent full-Unicode charset).
- **applies_to:** web-api, fullstack
- **signals:** `CHARSET=utf8\b|charset='utf8'|COLLATE utf8_` (not `utf8mb4`); DB config
  lacking a full-Unicode charset.
- **confirm:** warning if the DB/columns storing user text use a non-full-Unicode charset.
  pass for utf8mb4 / UTF-8 databases (Postgres default is fine).
- **severity:** Low (enterprise consumer app → Medium)
- **remediation:** Use `utf8mb4` charset + collation for text columns.
- **compliance_refs:** ISO25010 Adaptability

## Localization Readiness (advisory unless user-facing multi-locale)

### i18n_hardcoded_strings — User-facing text externalized
- **intent:** Hardcoded UI strings can't be translated; externalizing to resource
  bundles/catalogs is the foundation of localization.
- **applies_to:** web-api (rendered), fullstack, frontend-spa
- **signals:** literal user-facing strings in templates/components/handlers not wrapped in a
  translation call (`gettext|_\(|t\(|<Trans|i18n\.|{{ _(`); alert/label/button text inline.
- **confirm:** warning if user-facing strings are hardcoded while i18n intent exists. skip if
  single-locale-by-design. Never flag logs/errors/debug/machine strings.
- **severity:** Medium (no i18n intent → Info/skip)
- **remediation:** Move user-facing strings to translation catalogs; wrap in the i18n API.
- **compliance_refs:** ISO25010 Adaptability

### i18n_locale_formatting — Locale-aware date/number/currency
- **intent:** Hardcoded formats (`MM/DD/YYYY`, `$`, `.` decimals) are wrong for most locales.
- **applies_to:** fullstack, frontend-spa
- **signals:** hardcoded date format strings; `"$" +`/`"€" +` currency concat;
  `toFixed(2)` money without locale; manual `strftime("%m/%d/%Y")` for display.
- **confirm:** warning if dates/numbers/currency are formatted with fixed patterns instead of
  a locale-aware formatter (`Intl.*`, ICU, babel). pass for locale-aware formatting.
- **severity:** Medium (no i18n intent → Low/Info)
- **remediation:** Use `Intl.DateTimeFormat`/`Intl.NumberFormat`/ICU/babel per user locale.
- **compliance_refs:** ISO25010 Adaptability

### i18n_pluralization — Locale plural rules via i18n lib
- **intent:** `f"{n} items"` breaks for languages with multiple plural forms.
- **applies_to:** fullstack, frontend-spa
- **signals:** `f"{.*} (item|result|file)s"`; manual `if n == 1 ... else ...s` for UI text.
- **confirm:** warning if plural UI text is hand-built instead of using the i18n library's
  plural API. skip if no i18n intent.
- **severity:** Low
- **remediation:** Use the i18n plural API (ICU plural, `ngettext`, `Trans` count).
- **compliance_refs:** ISO25010 Adaptability

### i18n_string_concat — No sentence assembly by concatenation
- **intent:** Building sentences from fragments (`greeting + name + ", " + msg`) breaks word
  order and grammar in other languages.
- **applies_to:** fullstack, frontend-spa
- **signals:** user-facing sentences assembled via `+`/template concat of translated
  fragments + variables.
- **confirm:** warning if translatable sentences are concatenated from parts rather than using
  a single parameterized message. skip if no i18n intent.
- **severity:** Low
- **remediation:** Use one parameterized message string with named placeholders.
- **compliance_refs:** ISO25010 Adaptability

### i18n_rtl — RTL / bidirectional layout support
- **intent:** Hard-left/right layout breaks for Arabic/Hebrew; logical properties + `dir`
  adapt automatically.
- **applies_to:** frontend-spa
- **signals:** physical CSS (`margin-left|padding-right|text-align:\s*left|float:\s*left`)
  pervasively without logical equivalents; no `dir`/`[dir=rtl]` handling.
- **confirm:** warning (only under i18n intent) if layout uses physical directions with no
  RTL/logical-property support. skip for LTR-only products.
- **severity:** Low
- **remediation:** Use logical properties (`margin-inline-start`), set `dir`, test RTL.
- **compliance_refs:** ISO25010 Adaptability

### i18n_text_expansion — UI tolerates text expansion
- **intent:** Translations can be 30–200% longer; fixed-width/truncated UI clips them.
- **applies_to:** frontend-spa
- **signals:** fixed pixel widths on text containers; `overflow: hidden` + `white-space:
  nowrap` on labels/buttons holding translatable text.
- **confirm:** info (only under i18n intent) if UI assumes fixed text width for translatable
  strings.
- **severity:** Info (enterprise multi-locale → Low)
- **remediation:** Allow flexible widths / wrapping; test with pseudo-localization.
- **compliance_refs:** ISO25010 Adaptability
